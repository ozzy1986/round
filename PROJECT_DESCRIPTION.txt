1. WHAT IT IS
--------------
Interval timer for boxing, MMA, HIIT, running. Users define profiles: each profile has a name, an emoji, and an ordered list of rounds. Each round has a name (announced at start), duration in seconds, and an optional "warn 10 sec" (voice "X in 10 seconds" plus per-second tick in the last 10 seconds). One tap to start; app announces round names by voice and warnings. Works offline on the phone; syncs with the backend when online. Delivered as: backend API + Telegram bot (both run on the VPS), and an Android app that connects to the VPS server.


2. KEY FEATURES (from plan)
---------------------------
- Profiles: name, user-chosen emoji, list of rounds (order matters).
- Round: name (spoken at start), durationSeconds, warn10sec (voice "X in 10 sec" + tick each second).
- Quick entry: template "N x (work + rest)" generates N pairs (e.g. "3x5 1" = 3 x (5 min work + 1 min rest)); custom rounds (e.g. "5 run, 5 walk, 4 run, 15 walk").
- Timer screen: big countdown + current round emoji (profile emoji or round-based).
- i18n: EN + RU from start; strings in backend and Android resources.
- Offline-first: Android works without network; backend used for sync and Telegram.


3. ARCHITECTURE
---------------
- Server: API and Telegram bot run on the VPS (round.ozzy1986.com). The Android app and Telegram users connect to this server. The local machine is for development only (running backend/bot locally, emulator).
- Backend (this repo): Node.js, TypeScript, Fastify, PostgreSQL. REST API for profiles and rounds CRUD. Telegram bot (Telegraf) runs the same timer logic and sends TTS voice messages. Timer engine is a shared pure TS module (no I/O).
- Android app: in subfolder android/. Kotlin, Jetpack Compose, Room for local storage, Retrofit for sync with backend on VPS. Timer runs locally with same event contract; TTS and optional tick sounds.
- Shared timer contract: events emitted are round_start, tick (remainingSeconds), warn10, round_end, training_end. Backend uses the TS engine; Android reimplements the same logic in Kotlin.

6. DATA MODEL (PostgreSQL)
--------------------------
profiles:
  id (uuid, PK), name (text), emoji (text), user_id (text, nullable), created_at, updated_at (timestamptz)
  Index: (user_id, updated_at DESC) for list/get by user and delta sync.

rounds:
  id (uuid, PK), profile_id (uuid, FK -> profiles, CASCADE), name (text), duration_seconds (int), warn10sec (boolean), position (int)
  Index on profile_id; unique (profile_id, position). Trigger updates profile.updated_at when rounds change (for delta sync).

users:
  id (uuid, PK), is_anonymous (boolean), telegram_id (bigint, unique, nullable), created_at (timestamptz)

refresh_tokens:
  id (uuid, PK), token_hash (text), user_id (uuid, FK -> users, CASCADE), expires_at (timestamptz), revoked_at (timestamptz), created_at (timestamptz)

Auth: JWT-based. Access token 15m; refresh token 30 days, rotated on use. Anonymous registration (POST /auth/register) and Telegram auth (POST /auth/telegram with X-Bot-Secret) return token + refresh_token. POST /auth/refresh exchanges refresh_token for new token pair; POST /auth/logout revokes refresh_token. User isolation: all profile/round queries filter by user_id.


7. API ENDPOINTS
---------------
  GET    /profiles              - list profiles (query: limit?, cursor?). With include=rounds and optional updated_since= (ISO-8601): returns profiles with rounds in one response (batch + delta sync).
  GET    /profiles/:id          - get profile with rounds (404 if not found)
  POST   /profiles              - create profile (body: name, emoji)
  PATCH  /profiles/:id          - update profile (body: name?, emoji?)
  DELETE /profiles/:id          - delete profile (cascades to rounds)

  GET    /profiles/:id/rounds   - list rounds for profile (ordered by position)
  POST   /profiles/:id/rounds   - create round (body: name, duration_seconds, warn10sec?, position)
  PUT    /profiles/:id/rounds   - replace all rounds for profile (body: rounds: [{ name, duration_seconds, warn10sec?, position }]); atomic.
  PATCH  /rounds/:roundId      - update round
  DELETE /rounds/:roundId      - delete round

  POST   /auth/register        - create anonymous user; returns token, refresh_token, user_id
  POST   /auth/telegram        - (X-Bot-Secret) create/find user by telegram_id; returns token, refresh_token, user_id
  POST   /auth/refresh         - body: refresh_token; returns new token, refresh_token, user_id (rotation)
  POST   /auth/logout          - body: refresh_token; revokes token (204)

  GET    /                     - liveness (no DB); returns { status: 'ok' }
  GET    /health               - readiness (DB check); returns { status: 'ok' } or 503

All request/response bodies validated via Fastify JSON schema. UUIDs in path.


8. TIMER ENGINE (shared logic)
------------------------------
Input: TimerProfile { name, emoji, rounds: [{ name, durationSeconds, warn10sec }] }.
Output: stream of events (callback or advance() per second).

Events:
  round_start  - start of a round (roundIndex, round, totalRounds)
  tick         - every second (roundIndex, round, remainingSeconds, totalRounds)
  warn10       - when remainingSeconds === 10 and warn10sec (roundIndex, round, totalRounds)
  round_end    - end of round
  training_end - all rounds done

Pure logic; no I/O. Used by backend (Telegram) and by Android (Kotlin reimplementation).


9. TELEGRAM BOT
---------------
UI: Button-based (inline keyboards). No need to type commands; user taps buttons.

/start: Welcome message (i18n by user language_code) with two buttons: "Profiles", "Quick timer".
  - Profiles: Lists profiles from API as buttons (emoji + name); tap to start that profile's timer. "Back" returns to main menu.
  - Quick timer: Presets as buttons — e.g. 3×5+1, 5×3+1, "5 run, 5 walk". "Back" returns to main menu.
Commands (still supported): /profiles (same list with buttons), /timer <spec> (with no arg shows main menu; with arg starts timer).
<spec> for /timer: "3x5 1" (template), "5 run, 5 walk" (custom), or profile name/ID from API.

Flow: runs timer-engine in process; every second calls advance(). On round_start and warn10, synthesizes speech via Google TTS and sends voice message. On training_end sends text "Training finished." One running timer per user (stored in memory).

Env: TELEGRAM_BOT_TOKEN required. On VPS, API_URL is set so the bot talks to the local API (127.0.0.1:3001). Optional: GOOGLE_APPLICATION_CREDENTIALS for TTS.


10. ANDROID APP
---------------
Location: android/ (Gradle, Kotlin, Compose).

Screens: profile list, profile/round editor (name, emoji, rounds with name/duration/warn10 and label), timer. The UI uses Material 3 components, dynamic colors, elevated cards, and Material icons (`material-icons-extended`). Profile cards: emoji circle (72dp, 34sp) flush to left edge, Play button (72dp, 36dp icon) flush to right edge, card corner radius 36dp aligned with circles, text in the middle column with 14dp horizontal padding; ElevatedCard with surfaceVariant background. Cards show quick stats (round count + total duration), and Start is disabled for profiles without rounds. The editor uses Material 3 Switches for toggles and handles capitalization. Training (profile) name is limited to 30 characters; round name is limited to 20 characters (input and save). Round duration minimum is 5 seconds; on focus loss, values below 5 or empty auto-correct to 5. Emoji field limited to a single emoji (BreakIterator grapheme cluster). Each round has a numbered circle: tap to select/deselect, long-press on the circle or card padding to drag and reorder (haptic feedback, visual lift). Delete button on each round. When rounds are selected, a "Copy × N" action duplicates the block 1–99 times (e.g. one work + one rest copied 12 times for boxing, then delete the last rest). The timer features a large dynamic CircularProgressIndicator (thick stroke, 36dp), the profile emoji rendered at 120sp behind the countdown at 50% opacity, tabular numerals for the countdown, and a step indicator chip ("1 / N"). Timer screen top bar shows back button and profile name only (no emoji). The screen is kept ON (`FLAG_KEEP_SCREEN_ON`) during a running workout. Pause freezes the countdown (button becomes "Пауза" / "Pause"); when paused, two buttons appear: Resume ("Продолжить" / "Resume") to continue and Stop ("Стоп" / "Stop") to end the training. Timer button groups are exclusive (when{}): finished -> Restart, running+paused -> Resume+Stop, running -> Pause, else -> Start; prevents overlapping buttons during state transitions. A "Пауза" chip appears next to the step indicator when paused. When training finishes (naturally or via Stop), "Тренировка завершена" is shown and "Restart/Повторить" resets the timer. The profile list top bar includes a language selector chip (dropdown menu with 8 languages: EN, RU, UZ, KK, AZ, TG, TT, ZH); the preference is persisted in SharedPreferences and applied via attachBaseContext override on activity recreate. Profile editor: delete profile shows a confirmation dialog (RU/EN) before removal. All icon buttons have localized contentDescription for accessibility. "Add round" label is "Add round" / "Добавить раунд" (icon + text, no duplicate plus). Data in Room; sync with backend on VPS when online (Retrofit). Sync runs only when user saves a training (create/update/delete profile or save rounds) and when connectivity is re-established after being lost; on connectivity restore a 1-minute debounce ensures sync runs only after 1 minute of stable connection. Pull-to-refresh: on profile list, user pulls down to sync all profiles from server (forceSync, ignores throttle); on profile editor, pull syncs only that specific training (syncSingleProfile via GET /profiles/:id); on timer screen, pull syncs that training only (syncSingleProfile). Standard Android PullToRefreshContainer with spinner at top. When Start is disabled (new training, sync in progress, or TTS cache building), the timer screen shows a spinner and "Loading…" on the Start button until profile and TTS cache are ready. Sync uses GET /profiles?include=rounds (batch, one request per 100 profiles) and optional updated_since for delta; save rounds uses PUT /profiles/:id/rounds (atomic). Throttle: sync skipped if last sync was < 30 s ago. Auth: TokenStore (EncryptedSharedPreferences when available) holds access token and refresh_token; on 401 OkHttp Authenticator calls POST /auth/refresh via synchronous Call and retries with the new token. Timer: same event contract. TTS for round name and "X через 10 секунд" (Russian when device locale is RU) uses USAGE_ALARM stream to prevent volume ducking. One continuous prolonged tone (1200 ms sine wave via AudioTrack, USAGE_ALARM) at training start (first round) and at end of each non-final round. Training end: TTS "Тренировка [name] завершена" (no tone). Round names are shown in full on the timer display (up to 2 lines, ellipsis on overflow). Short ticks (150 ms, ToneGenerator TONE_CDMA_PIP, STREAM_ALARM) per second in the last 10 sec when "Предупредить за 10 сек" ON and duration >= 10 sec. App waits for TTS ready before first announcement. Accessibility: content descriptions are localized (EN/RU) and Android 14 back-invoked callback is enabled in manifest.

Performance optimizations: (1) App load — three-stage pipeline for fastest possible launch: (a) SplashScreen API (Theme.Round.Splash) shows instantly on tap; (b) MainActivity.onCreate uses a native FrameLayout placeholder (solid #1A1110 background, zero Compose overhead) as the first frame, so "Displayed" fires before Compose initializes; (c) Compose (RaundTheme + full UI) initializes via placeholder.post{} after the first frame, with reportFullyDrawn() when ready. Application.onCreate does zero main-thread work — no eager database init, no profileRepository access; all init (profileRepository, preloadCache, requestSync) runs on Dispatchers.IO via initScope.launch. repositoryReadyFlow (StateFlow) signals when preload completes; requestSync() runs in a separate coroutine. Before appReady, a custom AppLoadingScreen (gradient background, animated circular progress + "Loading…" text) is shown; after appReady, NavHost with pre-cached data renders with no double-recomposition. Manual baseline profile (app/src/main/baseline-prof.txt) covers Compose, Material3, Navigation, and Lifecycle hot paths; force-compile on device with `adb shell cmd package compile -m speed-profile -f com.raund.app`. Measured results (TECNO KL6, Android 14, release build): cold start avg 330ms / min 299ms (process creation 123ms + system window 170ms = 293ms floor, app code adds ~7ms); warm start avg 83ms / min 66ms (well under 100ms target). (1b) Timer screen: same single-training pull-to-refresh and sync-in-progress indicator on Start button as above. (2) TTS pre-caching uses parallel TextToSpeech instances (up to 3 concurrent) to synthesize missing phrases ~2-3x faster than sequential. (3) Profile save is local-first: Room insert returns instantly (<10ms), API call runs in a background coroutine (fire-and-forget); sync reconciles on next connectivity. (4) OkHttp has explicit timeouts (5s connect, 10s read/write) and a connection pool (5, 30s); an interceptor logs HTTP timing. (5) R8 minification enabled for release builds (isMinifyEnabled=true, isShrinkResources=true) with ProGuard rules for Retrofit, Gson, Room, Parcelize; reduces APK from ~20MB to ~1.5MB. (6) Notification updates are throttled: cached Notification.Builder, startForeground called every 5s, notify() for intermediate ticks. (7) Sync upserts wrapped in database.withTransaction{} for single SQLite commit per page. (8) MediaPlayer instance is reused across voice playbacks (reset instead of create/release). (9) Prolonged tone ShortArray buffer is computed once and cached for reuse across round transitions. (10) Timer state: TimerStateHolder singleton exposes StateFlow<TimerState>; TimerService updates it every tick (no per-second BroadcastReceiver); TimerScreen collects via collectAsState so only the countdown ring recomposes. TimerStateHolder.update() uses MutableStateFlow.update{} (compare-and-set) for thread-safe partial updates between timer thread and main thread. (11) ViewModels: ProfileListViewModel (combined profiles + roundStats flow), ProfileEditorViewModel (name, emoji, rounds as RoundEditState, selection, drag), TimerViewModel (profile, timer state, finished); state survives config changes. (12) ProfileEditorScreen rounds: LazyColumn with itemsIndexed for round cards (only visible items composed). (13) Drag-and-drop: shared Modifier.dragReorder in DragReorderModifier.kt used by round card and numbered circle. (14) TimerService audio: single-thread ExecutorService for playback; prolonged-tone AudioTrack cached (MODE_STATIC, reloadStaticData + play) instead of create/release per round; playProlongedTone returns immediately after play() (no Thread.sleep) so TTS cache can start without 1200ms delay — tone plays in hardware and stops on its own. (15) Silent catch blocks in ProfileRepository (ensureToken, syncFromApi) and TimerService (playCacheFile, playProlongedTone) log with Log.w. (16) SyncOnConnectivityEffect: pendingSyncJob is a plain var inside DisposableEffect (no mutableStateOf), avoiding recomposition when the job is set/cancelled.

Settings: gear icon on profile list top bar opens a dialog with two Switches (SettingsManager, SharedPreferences): (1) "Keep running when screen off" (default ON); when OFF, timer auto-pauses on ACTION_SCREEN_OFF, stays paused through lock screen, and auto-resumes only when the timer screen becomes visible (TIMER_VISIBLE). (2) "Keep running when leaving the app" (default ON); when OFF, timer auto-pauses when user navigates away (home/another app) while screen is still on, auto-resumes on TIMER_VISIBLE. A 300ms delay in TIMER_HIDDEN distinguishes screen-off from leaving-app (SCREEN_OFF cancels the delayed check); leaveAppCheckRunnable also checks PowerManager.isInteractive() directly so delayed ACTION_SCREEN_OFF broadcasts (TECNO HiOS) do not cause false leave-app detection. Manual pause/resume clears all auto-pause flags. Pause stores remaining ms; resume reconstructs roundStartRealtimeMs = now - (duration - remaining) so the countdown continues from the exact paused position. When the timer is running, TimerService is started as a foreground service (foregroundServiceType=mediaPlayback) and holds PARTIAL_WAKE_LOCK so the timer and audio (TTS, tones) continue with screen off; service stops when the timer ends or the user goes back. TimerService runs in the main process (no separate :timer process). Tick scheduling: background Thread loop with Thread.sleep(1000); PARTIAL_WAKE_LOCK keeps the CPU awake. For OEMs that freeze background threads (e.g. TECNO HiOS), the service uses a real continuous audio playback pipeline so the system treats the app like a music player (exempt from freeze): (1) AudioFocus AUDIOFOCUS_GAIN with USAGE_MEDIA/CONTENT_TYPE_MUSIC; (2) MediaSession active with PlaybackStateCompat.STATE_PLAYING; (3) a keep-alive AudioTrack in MODE_STREAM (USAGE_MEDIA) that runs for the whole timer duration, writing a low-amplitude sine wave in a dedicated thread so the audio pipeline is actively playing (like Spotify); foreground is kept while the timer runs (no stopForeground when the timer screen is visible). POST_NOTIFICATIONS is declared and requested at runtime on Android 13+; notification channel raund_timer_v3 (IMPORTANCE_DEFAULT) shows round name and countdown. Tapping the notification navigates to the running timer. onTaskRemoved stops the service when the user removes the app from recents. Permission: FOREGROUND_SERVICE_MEDIA_PLAYBACK.

Tests cover: TimerStateHolder (thread safety, update, reset), AuthAuthenticator (retry guard, refresh flow, error cases), TokenStore (CRUD, encryption fallback clears tokens), ProfileRepository (atomic saveRounds, sanitization, validation), BuildConfig (HTTPS URL, SENTRY_DSN field).

- WakeLock safety: TimerService acquires PARTIAL_WAKE_LOCK with a 4-hour timeout (MAX_WAKE_LOCK_MS) so the lock cannot leak if the process is killed without onDestroy. stopPreviousTimer() runs on a dedicated background executor (stopTimerExecutor) to avoid blocking the main thread and ANR risk.
- Auth retry guard: AuthAuthenticator returns null if the request already had an Authorization header, preventing infinite 401->refresh loops when the refreshed token is also rejected.
- Atomic saveRounds: ProfileRepository.saveRounds wraps roundDao.deleteByProfileId + roundDao.insertAll in database.withTransaction{} so a crash between delete and insert cannot lose data.

So the countdown reflects real wall-clock time, not thread wake count (avoids drift from GC, CPU load, thermal throttling). To prevent HiOS/Hiber from freezing the process, a continuous real playback pipeline runs for the whole timer: AudioTrack in MODE_STREAM (USAGE_MEDIA) writing a low-amplitude sine wave in a dedicated thread, plus MediaSession (active, STATE_PLAYING) and AudioFocus (AUDIOFOCUS_GAIN); the system sees active media playback and does not freeze. Timer and audio (TTS, ToneGenerator, prolonged tone) continue with screen off. All audio ops in try-catch. - Timer tick scheduling: background Thread loop; remaining seconds derived from elapsedRealtime() with ceiling division ((remainingMs+999)/1000) so Thread.sleep overshoot past a second boundary does not skip the displayed second; sleep until next second boundary (min 1ms, max 1000ms). Combined with PARTIAL_WAKE_LOCK + foreground service (mediaPlayback) + continuous keep-alive AudioTrack (MODE_STREAM, USAGE_MEDIA) + MediaSession + AudioFocus so HiOS does not freeze the process.
