CORE PRINCIPLES (MANDATORY)
---------------------------
1. Focus on security, speed, stability, scalability.
2. DRY, KISS, SOLID.
3. Test-driven development. All code should be covered with tests.


1. WHAT IT IS
--------------
Interval timer for boxing, MMA, HIIT, running. Users define profiles: each profile has a name, an emoji, and an ordered list of rounds. Each round has a name (announced at start), duration in seconds, and an optional "warn 10 sec" (per-second tick in the last 10 seconds). One tap to start; app announces round names by voice. Works offline on the phone; syncs with the backend when online. Delivered as: backend API + Telegram bot (both run on the VPS), and an Android app that connects to the VPS server.


2. KEY FEATURES (from plan)
---------------------------
- Profiles: name, user-chosen emoji, list of rounds (order matters).
- Round: name (spoken at start), durationSeconds, warn10sec (tick each second in the last 10 sec).
- Quick entry: template "N x (work + rest)" generates N pairs (e.g. "3x5 1" = 3 x (5 min work + 1 min rest)); custom rounds (e.g. "5 run, 5 walk, 4 run, 15 walk").
- Timer screen: big countdown (H:MM:SS when >= 1 hour, MM:SS otherwise) + current round emoji (profile emoji or round-based).
- i18n: EN + RU from start; strings in backend and Android resources.
- Offline-first: Android works without network; backend used for sync and Telegram.


3. ARCHITECTURE
---------------
- Server: API and Telegram bot run on the VPS (round.ozzy1986.com). The Android app and Telegram users connect to this server. The local machine is for development only (running backend/bot locally, emulator).
- Backend (this repo): Node.js, TypeScript, Fastify, PostgreSQL. REST API for profiles and rounds CRUD. Telegram bot (Telegraf) runs the same timer logic and sends TTS voice messages. Timer engine is a shared pure TS module (no I/O).
- Android app: in subfolder android/. Kotlin, Jetpack Compose, Room for local storage, Retrofit for sync with backend on VPS. Timer runs locally with same event contract; TTS and optional tick sounds.
- Shared timer contract: events emitted are round_start, tick (remainingSeconds), warn10, round_end, training_end. Backend uses the TS engine; Android reimplements the same logic in Kotlin.

6. DATA MODEL (PostgreSQL)
--------------------------
profiles:
  id (uuid, PK), name (text), emoji (text), user_id (text, nullable), created_at, updated_at (timestamptz)
  Index: (user_id, updated_at DESC) for list/get by user and delta sync.

rounds:
  id (uuid, PK), profile_id (uuid, FK -> profiles, CASCADE), name (text), duration_seconds (int), warn10sec (boolean), position (int)
  Index on profile_id; unique (profile_id, position). Trigger updates profile.updated_at when rounds change (for delta sync).

users:
  id (uuid, PK), is_anonymous (boolean), telegram_id (bigint, unique, nullable), created_at (timestamptz)

refresh_tokens:
  id (uuid, PK), token_hash (text), user_id (uuid, FK -> users, CASCADE), expires_at (timestamptz), revoked_at (timestamptz), created_at (timestamptz)

Auth: JWT-based. Access token 15m; refresh token 30 days, rotated on use. Anonymous registration (POST /auth/register) and Telegram auth (POST /auth/telegram with X-Bot-Secret) return token + refresh_token. POST /auth/refresh exchanges refresh_token for new token pair; POST /auth/logout revokes refresh_token. User isolation: all profile/round queries filter by user_id.


7. API ENDPOINTS
---------------
  GET    /profiles              - list profiles (query: limit?, cursor?). With include=rounds and optional updated_since= (ISO-8601): returns profiles with rounds in one response (batch + delta sync).
  GET    /profiles/:id          - get profile with rounds (404 if not found)
  POST   /profiles              - create profile (body: name, emoji)
  PATCH  /profiles/:id          - update profile (body: name?, emoji?)
  DELETE /profiles/:id          - delete profile (cascades to rounds)

  GET    /profiles/:id/rounds   - list rounds for profile (ordered by position)
  POST   /profiles/:id/rounds   - create round (body: name, duration_seconds, warn10sec?, position)
  PUT    /profiles/:id/rounds   - replace all rounds for profile (body: rounds: [{ name, duration_seconds, warn10sec?, position }]); atomic.
  PATCH  /rounds/:roundId      - update round
  DELETE /rounds/:roundId      - delete round

  POST   /auth/register        - create anonymous user; returns token, refresh_token, user_id
  POST   /auth/telegram        - (X-Bot-Secret) create/find user by telegram_id; returns token, refresh_token, user_id
  POST   /auth/refresh         - body: refresh_token; returns new token, refresh_token, user_id (rotation)
  POST   /auth/logout          - body: refresh_token; revokes token (204)

  GET    /                     - liveness (no DB); returns { status: 'ok' }
  GET    /health               - readiness (DB check); returns { status: 'ok' } or 503

All request/response bodies validated via Fastify JSON schema. UUIDs in path. Round duration_seconds: 0–7200 (POST/PATCH), 5–7200 (PUT). Max 30 rounds per profile (PUT maxItems).


8. TIMER ENGINE (shared logic)
------------------------------
Input: TimerProfile { name, emoji, rounds: [{ name, durationSeconds, warn10sec }] }.
Output: stream of events (callback or advance() per second).

Events:
  round_start  - start of a round (roundIndex, round, totalRounds)
  tick         - every second (roundIndex, round, remainingSeconds, totalRounds)
  warn10       - when remainingSeconds === 10 and warn10sec (roundIndex, round, totalRounds)
  round_end    - end of round
  training_end - all rounds done

Pure logic; no I/O. Used by backend (Telegram) and by Android (Kotlin reimplementation).


9. TELEGRAM BOT
---------------
UI: Button-based (inline keyboards). No need to type commands; user taps buttons.

/start: Welcome message (i18n by user language_code) with two buttons: "Profiles", "Quick timer".
  - Profiles: Lists profiles from API as buttons (emoji + name); tap to start that profile's timer. "Back" returns to main menu.
  - Quick timer: Presets as buttons — e.g. 3×5+1, 5×3+1, "5 run, 5 walk". "Back" returns to main menu.
Commands (still supported): /profiles (same list with buttons), /timer <spec> (with no arg shows main menu; with arg starts timer).
<spec> for /timer: "3x5 1" (template), "5 run, 5 walk" (custom), or profile name/ID from API.

Flow: runs timer-engine in process; every second calls advance(). On round_start and warn10, synthesizes speech via Google TTS and sends voice message. On training_end sends text "Training finished." One running timer per user (stored in memory).

Env: TELEGRAM_BOT_TOKEN required. On VPS, API_URL is set so the bot talks to the local API (127.0.0.1:3001). Optional: GOOGLE_APPLICATION_CREDENTIALS for TTS.


10. ANDROID APP
---------------
Location: android/ (Gradle, Kotlin, Compose).

Screens: profile list, profile/round editor (name, emoji, rounds with name/duration/warn10 and label), timer. The UI uses Material 3 components, dynamic colors, elevated cards, and Material icons (`material-icons-extended`). Profile cards: emoji circle (72dp, 34sp) flush to left edge, Play button (72dp, 36dp icon) flush to right edge, card corner radius 36dp aligned with circles, text in the middle column with 14dp horizontal padding; ElevatedCard with surfaceVariant background. Cards show quick stats (round count + total duration), and Start is disabled for profiles without rounds. The editor uses Material 3 Switches for toggles and handles capitalization. Training (profile) name is limited to 30 characters; round name is limited to 20 characters (input and save). Round duration and other positive-integer helper fields use a numeric-only keyboard hint (`KeyboardType.NumberPassword`) plus an explicit IME restart when focus changes between text and numeric fields, so OEM keyboards switch both ways without requiring the user to close and reopen the keyboard. Minimum is 5 seconds, maximum 7200 seconds (2 hours); on focus loss, values outside 5–7200 auto-correct to nearest bound. Maximum 30 rounds per profile; "Add round" button disables at the limit and "Copy × N" truncates at 30. Emoji field limited to a single emoji (BreakIterator grapheme cluster). Each round has a numbered circle: tap to select/deselect, long-press on the circle or card padding to drag and reorder (haptic feedback, visual lift). Delete button on each round. When rounds are selected, a "Copy × N" action duplicates the block 1–99 times (e.g. one work + one rest copied 12 times for boxing, then delete the last rest). The timer features a large dynamic CircularProgressIndicator (thick stroke, 36dp), the profile emoji rendered at 120sp behind the countdown at 50% opacity, tabular numerals for the countdown, and a step indicator chip ("1 / N"). The countdown ring stays in place; only the bottom timer control cluster is nudged 10 px downward for better visual balance. Countdown text font scales down when remaining time is >= 1 hour (55% of base size, min 28sp) so long durations (e.g. 1:00:00) fit inside the circle; when remaining drops below 1 hour the font returns to full size. Timer screen top bar shows back button and profile name only (no emoji). Timer exit is single-shot: after the user starts leaving the timer, additional toolbar taps and system-back events are ignored until the screen is disposed, preventing the rapid-back black screen. The screen is kept ON (`FLAG_KEEP_SCREEN_ON`) during a running workout. Pause freezes the countdown (button becomes "Пауза" / "Pause"); when paused, two buttons appear: Resume ("Продолжить" / "Resume") to continue and Stop ("Стоп" / "Stop") to end the training. Timer button groups are exclusive (when{}): finished -> Restart, running+paused -> Resume+Stop, running -> Pause, else -> Start; prevents overlapping buttons during state transitions. A "Пауза" chip appears next to the step indicator when paused. When training finishes (naturally or via Stop), "Тренировка завершена" is shown and "Restart/Повторить" resets the timer. The profile list has no in-app back button; BackHandler consumes back so the activity does not finish when the user taps the top-left or system back. The TopAppBar navigation icon slot is filled (empty Box) so taps there do not trigger system back. The profile list top bar includes a language selector chip (dropdown menu with 8 languages: EN, RU, UZ, KK, AZ, TG, TT, ZH); the preference is persisted in SharedPreferences and applied via attachBaseContext override on activity recreate. Profile editor: delete profile shows a confirmation dialog (RU/EN) before removal. All icon buttons have localized contentDescription for accessibility. "Add round" label is "Add round" / "Добавить раунд" (icon + text, no duplicate plus). Data in Room; sync with backend on VPS when online (Retrofit). Sync runs only when user saves a training (create/update/delete profile or save rounds) and when connectivity is re-established after being lost; on connectivity restore a 1-minute debounce ensures sync runs only after 1 minute of stable connection. Pull-to-refresh: on profile list, user pulls down to sync all profiles from server (forceSync, ignores throttle); on profile editor, pull syncs only that specific training (syncSingleProfile via GET /profiles/:id); on timer screen, pull syncs that training only (syncSingleProfile). Standard Android PullToRefreshContainer with spinner at top. When Start is disabled (new training, sync in progress, or TTS cache building), the timer screen shows a spinner and "Loading…" on the Start button until profile and TTS cache are ready. Sync uses GET /profiles?include=rounds (batch, one request per 100 profiles) and optional updated_since for delta; save rounds uses PUT /profiles/:id/rounds (atomic). Throttle: sync skipped if last sync was < 30 s ago. Auth: TokenStore uses EncryptedSharedPreferences when available and falls back to in-memory tokens only for the current process if encryption init fails (plain-text prefs are cleared and never reused). On 401, OkHttp Authenticator calls POST /auth/refresh via synchronous Call, retries the original authenticated request once with the new access token, and uses priorResponse counting to prevent infinite retry loops. AppDatabase singleton now uses double-checked locking so concurrent access cannot create duplicate Room instances. Timer: same event contract. TTS announces round names on round start and the training-finished message only. The 10-second warning phrase is no longer spoken; when "Предупредить за 10 сек" is ON and round duration >= 10 sec, the app plays slightly lower warning ticks only. One continuous prolonged tone (1200 ms sine wave via AudioTrack, 1046.5 Hz, USAGE_ALARM) at training start (first round) and at end of each non-final round. Training end: TTS "Тренировка [name] завершена" (no tone). Round names are shown in full on the timer display (up to 2 lines, ellipsis on overflow). Short ticks use a louder custom AudioTrack tone (150 ms, 1568 Hz, volume 0.9, USAGE_ALARM) per second in the last 10 sec when "Предупредить за 10 сек" ON and duration >= 10 sec. App waits for TTS ready before first announcement. The loading screen splash image is now static at 1.5x scale while the repository warms up. Accessibility: content descriptions are localized (EN/RU) and Android 14 back-invoked callback is enabled in manifest.

Performance optimizations: (1) App load — three-stage pipeline for fastest possible launch: (a) SplashScreen API (Theme.Round.Splash) shows instantly on tap; (b) MainActivity.onCreate uses a native FrameLayout placeholder (solid #1A1110 background, zero Compose overhead) as the first frame, so "Displayed" fires before Compose initializes; (c) Compose (RaundTheme + full UI) initializes via placeholder.post{} after the first frame, with reportFullyDrawn() only after `repositoryReadyFlow` signals that meaningful content is ready. Application.onCreate does zero main-thread work — no eager database init, no profileRepository access; all init (profileRepository, preloadCache, requestSync) runs on Dispatchers.IO via initScope.launch. repositoryReadyFlow (StateFlow) signals when preload completes; requestSync() runs in a separate coroutine. Before appReady, a custom AppLoadingScreen (gradient background, static 1.5x splash image + "Loading…" text) is shown; after appReady, NavHost with pre-cached data renders with no double-recomposition. Manual baseline profile (app/src/main/baseline-prof.txt) covers Compose, Material3, Navigation, and Lifecycle hot paths; force-compile on device with `adb shell cmd package compile -m speed-profile -f com.raund.app`. Measured results (TECNO KL6, Android 14, release build): cold start avg 330ms / min 299ms (process creation 123ms + system window 170ms = 293ms floor, app code adds ~7ms); warm start avg 83ms / min 66ms (well under 100ms target). A debug build on the same phone currently cold-starts around 1.34s, so phone QA now defaults to `localRelease` to avoid false performance regressions from debug-only overhead. (1b) Timer screen: same single-training pull-to-refresh and sync-in-progress indicator on Start button as above. (2) TTS pre-caching uses parallel TextToSpeech instances (up to 3 concurrent) to synthesize missing phrases ~2-3x faster than sequential. (3) Profile save is local-first: Room insert returns instantly (<10ms), API call runs in a background coroutine (fire-and-forget); sync reconciles on next connectivity. (4) OkHttp has explicit timeouts (5s connect, 10s read/write) and a connection pool (5, 30s); an interceptor logs HTTP timing in debug, while release/localRelease builds strip `android.util.Log` calls via ProGuard. (5) Release and localRelease builds keep R8/resource shrinking enabled. `scripts/build-apk.ps1` defaults to deterministic `assembleLocalRelease` with `--no-daemon`, `--no-parallel`, and `-Dkotlin.compiler.execution.strategy=in-process` so local optimized builds do not depend on flaky daemon state. Current measured local results on this PC: first full `assembleRelease` after Android changes was 7m44s; the next unchanged `assembleRelease` finished in 23s. ProGuard keeps were narrowed for Retrofit/OkHttp to let R8 strip more unused code safely, and no-DSN builds omit the Sentry Android SDK entirely. (6) Notification updates are throttled: cached Notification.Builder, startForeground called every 5s, notify() for intermediate ticks. (7) Sync upserts wrapped in database.withTransaction{} for single SQLite commit per page. (8) MediaPlayer instance is reused across voice playbacks (reset instead of create/release). (9) Tone buffers are cached by spec and reused across round transitions and warning ticks; both prolonged tones and the louder 10-second ticks use cached `AudioTrack` instances instead of recreating sound primitives on every event. (10) Timer state: TimerStateHolder singleton exposes StateFlow<TimerState>; TimerService updates it every tick (no per-second BroadcastReceiver); TimerScreen collects via collectAsState so only the countdown ring recomposes. TimerStateHolder.update() uses MutableStateFlow.update{} (compare-and-set) for thread-safe partial updates between timer thread and main thread. (11) ViewModels: ProfileListViewModel (combined profiles + roundStats flow), ProfileEditorViewModel (name, emoji, rounds as RoundEditState, selection, drag), TimerViewModel (profile, timer state, finished); state survives config changes. (12) ProfileEditorScreen rounds: LazyColumn with itemsIndexed keyed by stable UUID per RoundEditState (prevents focus loss during typing); only visible items composed. (13) Drag-and-drop: shared Modifier.dragReorder in DragReorderModifier.kt used by round card and numbered circle. (14) TimerService audio: single-thread ExecutorService for playback; prolonged-tone AudioTrack cached (MODE_STATIC, reloadStaticData + play) instead of create/release per round; playProlongedTone returns immediately after play() (no Thread.sleep) so TTS cache can start without 1200ms delay — tone plays in hardware and stops on its own. (15) Silent catch blocks in ProfileRepository (ensureToken, syncFromApi) and TimerService (playCacheFile, playProlongedTone, playTickTone) log with Log.w. (16) SyncOnConnectivityEffect: pendingSyncJob is a plain var inside DisposableEffect (no mutableStateOf), avoiding recomposition when the job is set/cancelled.

Settings: gear icon on profile list top bar opens a dialog with two Switches (SettingsManager, SharedPreferences): (1) "Keep running when screen off" (default ON); when OFF, timer auto-pauses on ACTION_SCREEN_OFF and resumes after unlock when background running is allowed. (2) "Keep running when leaving the app" (default ON); when OFF, timer auto-pauses when user navigates away (home/another app) while screen is still on, auto-resumes on TIMER_VISIBLE. A 300ms delay in TIMER_HIDDEN distinguishes screen-off from leaving-app (SCREEN_OFF cancels the delayed check); leaveAppCheckRunnable also checks PowerManager.isInteractive() directly so delayed ACTION_SCREEN_OFF broadcasts (TECNO HiOS) do not cause false leave-app detection. For the mixed mode "screen off = pause, leaving app = continue", TimerService listens for `ACTION_USER_PRESENT` and also checks `ACTION_SCREEN_ON` when no keyguard is active. On Android 13+/14 the system screen-state receiver is registered with `RECEIVER_EXPORTED`, which restores delivery of the unlock broadcasts while the app stays in the background on TECNO HiOS, so the workout resumes after unlock without forcing the timer screen visible. Manual pause/resume clears all auto-pause flags. Pause stores remaining ms; resume reconstructs roundStartRealtimeMs = now - (duration - remaining) so the countdown continues from the exact paused position. When the timer is running, TimerService is started as a foreground service (foregroundServiceType=mediaPlayback) and holds PARTIAL_WAKE_LOCK so the timer and audio (TTS, tones) continue with screen off; service stops when the timer ends or the user goes back. TimerService runs in the main process (no separate :timer process). Tick scheduling: background Thread loop with Thread.sleep(1000); PARTIAL_WAKE_LOCK keeps the CPU awake. For OEMs that freeze background threads (e.g. TECNO HiOS), the service uses a real continuous audio playback pipeline so the system treats the app like a music player (exempt from freeze): (1) AudioFocus AUDIOFOCUS_GAIN with USAGE_MEDIA/CONTENT_TYPE_MUSIC; (2) MediaSession active with PlaybackStateCompat.STATE_PLAYING; (3) a keep-alive AudioTrack in MODE_STREAM (USAGE_MEDIA) that runs for the whole timer duration, writing a low-amplitude sine wave in a dedicated thread so the audio pipeline is actively playing (like Spotify); foreground is kept while the timer runs (no stopForeground when the timer screen is visible). POST_NOTIFICATIONS is declared and requested at runtime on Android 13+; notification channel raund_timer_v3 (IMPORTANCE_DEFAULT). Notification title is the full training (profile) name; content is the full current round name plus the countdown timer (e.g. "RoundName 1:23:45"). Tapping the notification navigates to the running timer. onTaskRemoved stops the service when the user removes the app from recents. Permission: FOREGROUND_SERVICE_MEDIA_PLAYBACK.

14. QUALITY & SECURITY
-----------------------
Aligned with CORE PRINCIPLES above: security, speed, stability, scalability; DRY, KISS, SOLID; TDD with full test coverage.
- Validate all input (Fastify schemas), parameterized queries only, no secrets in logs, .env not committed.
- CORS: configurable via CORS_ORIGINS env var (comma-separated origins; empty = deny in production, allow all in local dev).
- BOT_SECRET comparison uses crypto.timingSafeEqual (SHA-256 digest) to prevent timing attacks.
- Nginx: HTTP 8080 redirects to HTTPS 443. HSTS, X-Content-Type-Options, X-Frame-Options headers. TLS 1.2+. Upstream keepalive and gzip compression for JSON.
- Graceful shutdown: SIGTERM/SIGINT handlers drain in-flight requests, flush Sentry, and close DB pool/Redis before exit; shutdown is bounded by SHUTDOWN_TIMEOUT_MS (default 10000) so deploys do not hang forever on stuck dependencies.
- N+1 queries eliminated: getProfileWithRoundsById uses single query with json_agg; round update/delete use JOINed ownership checks (1 query instead of 2-3).
- Refresh token rotation: short-lived access token (15m), refresh token (30d) stored hashed in DB; rotation on /auth/refresh; Android retries on 401 via refresh.
- Redis (optional): when REDIS_URL set, rate limit is shared across PM2 workers; bot token/fileId/TTS caches use Redis with TTL. Required at scale (multi-instance). Rate limit key: verified JWT user id when Bearer is valid, else IP; invalid/forged bearer tokens fall back to IP; RATE_LIMIT_MAX (default 200), RATE_LIMIT_WINDOW (default 1 minute).
- PgBouncer (optional): deploy/pgbouncer.ini.sample; PG_POOL_MAX per worker (default 10); pool.on('error') and exhaustion warning; PG_STMT_TIMEOUT_MS is optional but production falls back to 10000 when unset.
- Sentry (optional): SENTRY_DSN for 5xx error reporting; health-check.sh for uptime cron. Android: the Sentry SDK is packaged only when SENTRY_DSN is provided for the build; auto-init disabled (meta-data io.sentry.auto-init=false); RaundApplication initializes it only when BuildConfig.SENTRY_ENABLED is true, so no-DSN builds omit the SDK/provider size cost. ProGuard keeps io.sentry.** and LineNumberTable/SourceFile only matter when the dependency is present.
- Android unit tests: JUnit 4, MockK, Robolectric, kotlinx-coroutines-test; test directory android/app/src/test/. Current count: 44 tests. Coverage includes: TimerStateHolder (thread safety, update, reset), TimerAutoResumePolicy (screen-off pause/unlock resume decisions), AuthAuthenticator (authenticated 401 refresh, retry guard, error cases), TokenStore (CRUD, encrypted storage fallback to memory-only, no plain-text persistence), ProfileRepository (atomic saveRounds, sanitization, validation), NavControllerExtensions (repeat-safe return to `profiles`), BuildConfig (HTTPS URL, SENTRY_DSN, SENTRY_ENABLED fields).
- Shared timer core; Vitest with coverage (timer-engine 100%, auth, cache, others high).
- Offline-first and data validation. UI: minimal, big countdown (shared formatDuration: H:MM:SS when >= 1h, MM:SS otherwise — used in profile list, timer screen, notification), clear typography.
- Timer tick sounds: 150ms custom 1568Hz AudioTrack tone at volume 0.9 on USAGE_ALARM for the last 10 seconds; no 10-second warning voice; TTS at volume 1.0f for round starts/training end; prolonged alarm tone at 1046.5Hz / volume 0.55f via AudioTrack.
- WakeLock safety: TimerService acquires PARTIAL_WAKE_LOCK with a 4-hour timeout (MAX_WAKE_LOCK_MS) so the lock cannot leak if the process is killed without onDestroy. stopPreviousTimer() runs on a dedicated background executor (stopTimerExecutor) to avoid blocking the main thread and ANR risk.
- Auth retry guard: AuthAuthenticator refreshes authenticated 401 responses once and uses priorResponse counting to stop infinite 401->refresh loops when the refreshed token is also rejected.
- Atomic saveRounds: ProfileRepository.saveRounds wraps roundDao.deleteByProfileId + roundDao.insertAll in database.withTransaction{} so a crash between delete and insert cannot lose data.
- Android manifest: allowBackup=false (prevents token extraction via adb backup), usesCleartextTraffic=false (forbids accidental HTTP).
- Room schema export: exportSchema=true in @Database; KSP arg room.schemaLocation=$projectDir/schemas for migration support.
- TokenStore fallback: if EncryptedSharedPreferences fails, logs error, clears any existing plain-text tokens, and keeps tokens in memory only for the current process instead of writing them to plain SharedPreferences.
- 152-FZ data consent: DataConsentPrefs stores consent flag; first-launch ConsentDialog (localized) explains anonymous ID and training sync, links to https://round.ozzy1986.com/privacy; ProfileRepository.ensureToken() and thus all API/sync calls are no-op until consent granted.
- Release signing: build.gradle.kts defines a release signingConfig from project properties (RELEASE_STORE_FILE, RELEASE_STORE_PASSWORD, RELEASE_KEY_ALIAS, RELEASE_KEY_PASSWORD); release tasks fail fast when the keystore is missing unless `-PALLOW_DEBUG_SIGNED_RELEASE=true` is passed explicitly for non-production builds. A separate deterministic `localRelease` build type uses the debug keystore, keeps release optimizations enabled, and is the default path for local phone QA. Generate keystore: scripts/generate-release-keystore.ps1 (requires JAVA_HOME). android/keystore/ in .gitignore.
- Screen-off handling: TimerService runs as foreground (mediaPlayback type), holds PARTIAL_WAKE_LOCK, and drives the timer from SystemClock.elapsedRealtime() (remaining = roundStartRealtime + durationMs - elapsedRealtime(); sleep until next second boundary). So the countdown reflects real wall-clock time, not thread wake count (avoids drift from GC, CPU load, thermal throttling). To prevent HiOS/Hiber from freezing the process, a continuous real playback pipeline runs for the whole timer: AudioTrack in MODE_STREAM (USAGE_MEDIA) writing a low-amplitude sine wave in a dedicated thread, plus MediaSession (active, STATE_PLAYING) and AudioFocus (AUDIOFOCUS_GAIN); the system sees active media playback and does not freeze. When screen-off auto-pause is enabled but leaving-app continue is also enabled, the service resumes on unlock via `ACTION_USER_PRESENT` / `ACTION_SCREEN_ON` (no-keyguard case). On Android 13+/14 the screen-state receiver is registered exported so those unlock broadcasts reach the background timer service on devices like TECNO HiOS. Timer and audio (TTS, custom warning ticks, prolonged tone) continue with screen off when that setting is enabled. All audio ops in try-catch. Permissions: WAKE_LOCK, FOREGROUND_SERVICE, FOREGROUND_SERVICE_MEDIA_PLAYBACK, POST_NOTIFICATIONS (runtime on Android 13+).
- Theme: orange primary (#FF8A00 dark / #E65100 light), purple secondary (#CE93D8 dark / #7B1FA2 light), warm orange tertiary (#FFB74D / #FF9800); bold typography; dynamic colors disabled (branded orange/purple palette enforced). - Timer tick scheduling: background Thread loop; remaining seconds derived from elapsedRealtime() with ceiling division ((remainingMs+999)/1000) so Thread.sleep overshoot past a second boundary does not skip the displayed second; sleep until next second boundary (min 1ms, max 1000ms). Combined with PARTIAL_WAKE_LOCK + foreground service (mediaPlayback) + continuous keep-alive AudioTrack (MODE_STREAM, USAGE_MEDIA) + MediaSession + AudioFocus so HiOS does not freeze the process.
